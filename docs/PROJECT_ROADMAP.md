# ainame 一年整体研发规划

## 1. 年度目标

第一年的核心目标是面向商业品牌命名用户，在测试/演示环境完成以下准商业闭环：

> AI 命名 → 域名与商标校验 → 品牌视觉生成 → 会员配额 → 支付宝模拟支付 → 数字资产沉淀 → 专家服务

支付宝支付仅用于模拟订单、权益开通、退款和异常补偿流程，不产生真实交易。因此，第一年的验收目标是验证产品、技术和运营闭环，不以真实收入作为验收条件。

## 2. 范围边界

### 2.1 正式建设能力

- 商业品牌 AI 命名与结构化分析报告。
- 域名可用性校验。
- Logo、Slogan、名片等品牌视觉资产生成。
- 会员套餐、每日配额、模拟订单和权益管理。
- 名称、报告、Logo、名片等数字资产管理。
- 用户、套餐、订单、任务、AI 配置和内容审计后台。
- 基于同一套 uni-app 工程交付 H5、微信小程序和 App。

### 2.2 试点能力

- 商标注册风险查询：接入一家合规数据供应商，只提供风险预估，不作为法律意见。
- Marketplace：后半年邀请少量品牌顾问试运营，不开放未经审核的专家自由入驻。

### 2.3 第一年度暂缓能力

- 微信公众号、抖音、小红书等社交媒体自动查重。
- 真实支付宝支付、分账、发票、对账和资金结算。
- 完整的 B 端 API PaaS、开发者 Dashboard 和正式 SLA。
- 分销佣金、合伙人结算和公开裂变活动。
- 宝宝八字五行等需要额外专业规则和合规评估的深度能力。

## 3. 当前实现基线

仓库目前已具备以下实现，后续应在现有代码上整合，不重复建设：

- FastAPI 认证、注册、登录和双 Token 基础能力。
- 基于 LangGraph、DeepSeek 的人名、企业名、宠物名生成与反馈迭代。
- 企业名称 `.com` 域名 WHOIS 查询。
- RAG 文件上传、ChromaDB 检索和异步处理基础。
- 品牌视觉和品牌方案生成基础。
- 会员套餐、每日命名/视觉配额和模拟订单模型。
- 名称与视觉资产库。
- 社区发布、投票、评论和举报功能。
- 专家申请、服务包、服务订单、报告和评价功能。
- 管理员用户、财务、AI 配置、知识库和审计接口。
- 部分会员、Marketplace、社区和视觉持久化自动化测试。

当前主要缺口是业务逻辑尚未完整下沉到 Service 层，命名等慢请求仍缺少统一任务平台，业务数据库仍需统一至 PostgreSQL，数字校验、生产级存储、可观测性和端到端发布体系尚未形成。

## 4. 目标架构

### 4.1 调用链

```text
H5 / 微信小程序 / App / 管理后台
                  │
                Nginx
                  │
          FastAPI Routers
                  │
              Services
          ┌───────┴────────┐
          │                │
     PostgreSQL       Redis + ARQ
                           │
                         Worker
              ┌────────────┼────────────┐
              │            │            │
         DeepSeek      校验供应商     图像供应商
```

### 4.2 分层职责

- `routers/`：只负责认证、参数校验、调用 Service 和响应映射。
- `services/`：负责命名任务、配额、订单、权益、Marketplace 等业务规则和事务边界。
- `repository/`：只负责 PostgreSQL 数据访问，不包含业务状态流转。
- `core/`：保留 LangGraph、RAG、图像生成和外部供应商适配等核心技术能力。
- Worker：执行命名、视觉、RAG、邮件和外部校验等慢任务。

### 4.3 数据与基础设施

- PostgreSQL 作为唯一业务数据库，使用 JSONB 保存命名报告、校验快照、模型输出和专家报告。
- LangGraph 检查点使用同一 PostgreSQL 集群中的独立 schema，避免与业务表混用。
- Redis 用于 ARQ 队列、验证码、缓存、限流和短期任务进度。
- 将现有 RabbitMQ RAG 队列和 FastAPI `BackgroundTasks` 分阶段迁移至 Redis + ARQ。
- Redis 队列按 `naming`、`visual`、`rag`、`mail` 划分，避免慢视觉任务阻塞命名请求。
- 图片和上传文件使用 S3 兼容对象存储；本地目录仅供开发环境使用。
- 第一年度继续使用 ChromaDB，但必须具备持久卷、备份、知识库版本和重建机制。
- 密钥、数据库连接和供应商配置全部通过环境变量加载，不写入源码或文档。

## 5. 核心接口与状态模型

新接口统一使用 `/api/v1` 前缀，旧接口保留一个版本周期供前端迁移。

### 5.1 主要接口

| 接口 | 用途 |
| --- | --- |
| `POST /api/v1/naming/jobs` | 提交命名任务，返回 `202` 和 `job_id` |
| `GET /api/v1/jobs/{job_id}` | 查询命名、视觉、RAG 或校验任务状态 |
| `POST /api/v1/names/{asset_id}/validations` | 触发域名和商标风险校验 |
| `POST /api/v1/brand-kits` | 基于已保存名称创建品牌方案 |
| `GET /api/v1/me/assets` | 查询名称、报告和视觉资产 |
| `POST /api/v1/membership/orders` | 创建会员模拟订单 |
| `POST /api/v1/mock-payments/{order_id}/complete` | 在测试环境完成支付宝模拟支付 |

前端以任务查询接口作为可靠状态来源；WebSocket 仅用于优化实时体验，断线后必须能够通过轮询恢复。

### 5.2 统一任务状态

```text
PENDING → QUEUED → RUNNING → SUCCEEDED
                           ├→ FAILED
                           └→ CANCELLED
```

任务必须记录任务类型、用户、输入摘要、状态、尝试次数、错误码、结果引用和时间戳。

### 5.3 业务一致性

- 创建任务和扣减配额必须支持幂等键。
- Worker 重试不得重复扣减配额、创建订单或生成重复资产。
- 任务最终失败后自动退还配额，并写入补偿与审计记录。
- Marketplace 的接单、拒单、交付、完成和退款只能通过 Service 执行状态转换。
- 模拟支付接口必须受环境开关和测试权限限制，并在界面明确标注“不产生真实交易”。

## 6. 十二个月路线图

### Q1：稳定基线与基础设施（M1–M3）

#### M1：基线收口

- 整理现有会员、视觉、社区、Marketplace 和管理端实现，建立功能与接口清单。
- 验证 Alembic 迁移链，补充新建数据库和已有数据库升级测试。
- 明确品牌命名输入、报告结构、会员套餐和专家服务标准。
- 修复源码编码问题，移除公开调试接口，建立统一错误码和请求 ID。

完成标准：现有核心测试稳定通过；接口、数据库表和未完成能力均有明确清单；不再继续扩展未验收模块。

#### M2：服务分层与 PostgreSQL 迁移

- 建立 Service 层，将 Router 中的配额、订单和 Marketplace 状态流转迁出。
- 将 MySQL 业务数据迁移至 PostgreSQL，提供数据迁移、校验和回滚步骤。
- 引入 `/api/v1`，旧接口保持兼容并标记废弃周期。

完成标准：测试环境完全使用 PostgreSQL；迁移前后用户、订单、权益和资产数量一致；关键业务不再由 Router 直接操作状态。

#### M3：统一异步任务平台

- 引入 Redis + ARQ，先迁移命名和 RAG，再迁移视觉和邮件。
- 实现统一任务表、任务查询接口、超时、重试、取消和失败补偿。
- 增加队列长度、处理延迟、失败率和 Worker 存活监控。

完成标准：API 提交慢任务后立即返回；Worker 重启后任务可以恢复；重复投递不会重复扣减权益。

### Q2：核心产品闭环（M4–M6）

#### M4：商业品牌命名质量

- 输出名称、寓意、典故、读音、行业匹配、品牌调性、负面联想和模型版本。
- 建立覆盖科技、餐饮、教育、母婴和游戏行业的固定评测集。
- 对 Prompt 和模型配置实行版本管理，每次发布保留回滚版本。

完成标准：结构化输出解析成功率不低于 99%；模型变更通过固定评测集；失败输出可追踪到 Prompt 和模型版本。

#### M5：数字资产校验

- 接入稳定域名供应商，生成全拼、首字母及 `.com`、`.cn`、`.ai` 组合。
- 接入一家商标数据供应商，根据类别、近似项和注册状态给出低、中、高风险。
- 为供应商请求增加超时、重试、缓存、熔断和降级提示。

完成标准：域名查询可用于演示环境；商标结果包含数据时间和风险声明；供应商故障不会阻断命名主流程。

#### M6：视觉、会员与模拟订单

- 完成 Logo、Slogan、名片品牌包的异步生成和资产持久化。
- 打通会员套餐、每日配额、模拟订单、支付宝模拟支付、权益开通和退款补偿。
- 用户可将选定名称、报告和视觉成果统一保存到资产库。

完成标准：用户可完整走通命名、校验、选名、视觉、模拟升级和资产保存；任务失败时权益补偿准确。

### Q3：全端发布与运营后台（M7–M9）

#### M7：全端功能收口

- uni-app 统一实现登录、命名、任务进度、品牌包、资产库、会员和模拟支付。
- 分别处理 H5、微信小程序和 App 的上传、存储、分享、网络恢复和资源兼容差异。

完成标准：三端使用同一后端接口完成核心用户路径；每端均有独立回归测试结果。

#### M8：管理后台与合规

- 补齐用户、套餐、订单、任务、供应商调用、配额、内容审计和异常补偿后台。
- Prompt 工作台支持草稿、测试、发布和回滚，禁止直接覆盖已发布版本。
- 完成敏感词、提示词注入防护、生成内容巡检和用户数据删除流程。

完成标准：运营人员无需操作数据库即可处理用户、权益、任务和内容问题；关键后台操作均有审计记录。

#### M9：测试/演示版本发布

- 部署测试/演示环境，明确展示模拟支付和商标风险声明。
- 建立模型成本、任务延迟、成功率、用户选名率和品牌包采用率看板。
- 收集首批真实用户行为和反馈。

完成标准：核心路径连续稳定运行两周；严重故障可定位、告警和补偿；不发生真实资金交易。

### Q4：Marketplace 试运营与质量收口（M10–M12）

#### M10：专家交付闭环

- 补齐专家资质审核、服务包、接单、拒单、报告草稿、交付和评价。
- AI 可以生成专家报告草稿，但专家提交前必须人工确认。
- 保存 AI 原始版本、人工修改结果和操作记录。

完成标准：专家订单从创建到报告交付全程可追踪；越权用户不能查看或修改订单和报告。

#### M11：邀请试运营

- 邀请少量品牌顾问和测试用户试运营，不开放无审核入驻。
- 增加申诉、管理员仲裁和模拟退款流程。
- 统计专家接单率、交付周期、报告满意度和问题类型。

完成标准：完成至少一轮端到端专家服务演练；异常订单能够由管理员闭环处理。

#### M12：工程质量收口

- 执行容量测试、数据库备份恢复、队列积压、供应商故障和 Worker 崩溃演练。
- 复盘命名质量、视觉采用率、任务失败和用户流失步骤。
- 根据数据选择第二年度优先建设 B 端 API PaaS 或邀请增长体系。

完成标准：年度验收指标达成；形成故障演练记录、遗留风险清单和下一年度立项依据。

## 8. 测试与验收

### 8.1 自动化测试范围

- Service 和 Repository 单元测试。
- PostgreSQL、Redis、ARQ Worker 和对象存储集成测试。
- DeepSeek、图像和校验供应商的 Mock 与契约测试；CI 不调用付费服务。
- 配额并发、任务重复投递、Worker 崩溃、供应商超时和重复支付回调测试。
- Alembic 全新建库、已有数据库升级和降级验证。
- H5、微信小程序和 App 核心路径回归测试。
- 管理员、普通用户、专家之间的权限隔离测试。

### 8.2 年度验收指标

- 任务提交接口 P95 响应时间小于 500ms。
- 命名任务 P95 完成时间小于 30 秒，成功率不低于 95%。
- 视觉任务成功率不低于 90%，最终失败后的配额补偿准确率为 100%。
- 并发与重试场景下不产生重复订单、重复扣减或重复资产。
- 核心交易、权益和任务状态流转测试覆盖率为 100%，项目整体测试覆盖率不低于 70%。
- Prompt 或模型变更必须通过固定评测集，不允许仅凭人工观感发布。
- PostgreSQL 备份恢复后，用户、订单、权益、任务和资产关系保持完整。

## 9. 运营与监控指标

- 产品指标：命名任务完成率、候选名称收藏率、校验触发率、品牌包生成率、模拟升级转化率。
- AI 指标：结构化输出成功率、评测得分、平均 Token 成本、模型错误率。
- 系统指标：API 延迟、队列长度、任务等待时间、Worker 成功率、供应商错误率。
- Marketplace 指标：专家接单率、平均交付周期、报告完成率、评价和申诉率。

任何核心指标异常必须能够通过请求 ID、任务 ID、用户 ID、模型版本和供应商调用记录完成定位。

## 10. 默认假设

- 第一年度核心用户是商业品牌命名客户。
- C 端采用会员套餐和每日配额，不引入点数钱包。
- 支付宝仅用于测试/演示环境的模拟支付。
- 商标查询只提供风险预估；社交媒体查重不承诺自动化结果。
- Marketplace 是后半年唯一重点扩展模块。
- B 端 API 仅预留租户、计量和幂等设计，不建设正式开发者 Dashboard。
- Growth 第一年度只预留分享来源和邀请关系数据，不进行佣金结算。
- 第一年度不包含真实支付、公开商业结算和生产发布承诺。
